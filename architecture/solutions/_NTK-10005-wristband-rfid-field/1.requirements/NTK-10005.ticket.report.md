## Description

The new RFID-enabled wristbands being rolled out for the 2026 summer season have two identifiers:

1. **wristband_id** - The visual alphanumeric code printed on the band (existing field)
2. **rfid_tag** - The RFID chip identifier readable by NFC/RFID scanners (new field needed)

Currently, the `CheckIn` schema in svc-check-in only has `wristband_id`. Guests must type or have staff manually enter the wristband ID at check-in kiosks. With the new RFID wristbands, kiosks equipped with RFID readers could scan the wristband instead, but there is no field in the schema to store or look up the RFID tag value.

We need to add an `rfid_tag` field to both the `CheckIn` and `WristbandAssignment` schemas so that:

- Kiosks can scan and record the RFID tag during check-in
- The system can look up a guest's check-in record by either wristband_id or rfid_tag
- Downstream services consuming check-in events receive the RFID tag for contactless interactions throughout the park

## Acceptance Criteria

1. The `CheckIn` schema in svc-check-in includes an optional `rfid_tag` field (string, max 64 characters)
2. The `WristbandAssignment` schema in svc-check-in includes an optional `rfid_tag` field matching the same format
3. The `GET /api/v1/checkins` endpoint supports filtering by `rfid_tag` query parameter
