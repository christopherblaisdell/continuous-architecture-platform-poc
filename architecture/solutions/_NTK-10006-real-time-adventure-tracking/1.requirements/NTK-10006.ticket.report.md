## User Story

> As a park safety officer,
> I want real-time GPS tracking of all guests on active adventures with automated emergency alerting,
> So that rescue response times are reduced and insurance tracking mandates are met.

## Use Cases

### UC-1: Activate Tracking on Check-In Completion

**Preconditions:** Guest has completed the check-in flow (status COMPLETE) and has an RFID wristband assigned.

**Main Flow:**
1. Guest completes check-in, triggering the `checkin.completed` event (which already carries `rfid_tag`)
2. svc-adventure-tracking receives the event and creates an active tracking session
3. The guest's wristband begins transmitting GPS coordinates at a frequency determined by adventure pattern (Pattern 1: 60s, Pattern 2: 30s, Pattern 3: 10s)
4. svc-adventure-tracking stores position updates and makes them available to the ops dashboard in real time

**Postconditions:** Guest position is visible on the live tracking map in web-ops-dashboard.

### UC-2: Monitor Guest Position Against Geofences

**Preconditions:** Tracking session is active; geofences are defined for the adventure route.

**Main Flow:**
1. svc-adventure-tracking receives a GPS position update from a wristband
2. The service evaluates the position against adventure-specific geofences (trail boundaries, restricted zones, emergency assembly points)
3. If the guest exits the defined geofence, a `geofence.breach` event is emitted
4. svc-notifications delivers an alert to the assigned guide and operations staff

**Postconditions:** Operations staff and guide are aware of the geofence breach.

### UC-3: Guest Triggers SOS

**Preconditions:** Guest is on an active adventure with a tracking-enabled wristband.

**Main Flow:**
1. Guest presses the SOS button on their wristband (or triggers SOS via the mobile app)
2. svc-adventure-tracking receives the SOS signal with the guest's current GPS coordinates
3. svc-adventure-tracking forwards the emergency to svc-emergency-response with guest identity, location, adventure details, and nearby guide information
4. svc-emergency-response creates an emergency record, dispatches the nearest rescue team, and triggers multi-channel emergency notifications
5. svc-notifications sends URGENT alerts to: the assigned guide, operations staff, the nearest ranger station, and emergency contacts on file

**Postconditions:** Emergency record exists, rescue team dispatched, all relevant parties notified within 60 seconds of SOS trigger.

### UC-4: Weather-Triggered Emergency Alerting

**Preconditions:** svc-weather detects a severe weather event (WARNING or EMERGENCY severity).

**Main Flow:**
1. svc-weather emits a weather alert event (SEVERE_STORM, FLASH_FLOOD, AVALANCHE_WARNING, etc.)
2. svc-adventure-tracking identifies all active tracking sessions in the affected area using geospatial queries
3. svc-emergency-response evaluates the alert severity and affected guest count
4. If evacuation is required, svc-emergency-response creates an emergency record and triggers bulk evacuation notifications
5. svc-notifications uses the bulk endpoint to alert all affected guests, guides, and operations staff simultaneously

**Postconditions:** All affected parties notified; emergency record created if evacuation required.

### UC-5: Deactivate Tracking on Adventure Completion

**Preconditions:** Guest's adventure is complete; reservation status changes to COMPLETED.

**Main Flow:**
1. svc-reservations emits a `reservation.status-changed` event with status COMPLETED
2. svc-adventure-tracking receives the event and terminates the active tracking session
3. The tracking session data is retained for insurance compliance (90-day retention) and then archived
4. Guest position is removed from the live tracking map

**Postconditions:** Tracking session closed; historical data retained per insurance mandate.

## Acceptance Criteria

1. GPS position updates are received and stored at the frequency matching the adventure's safety pattern (Pattern 1: 60s, Pattern 2: 30s, Pattern 3: 10s)
2. Tracking activates automatically when check-in completes — no manual trigger required
3. SOS alerts reach svc-emergency-response within 5 seconds of trigger
4. Emergency notifications are delivered to all relevant parties within 60 seconds of SOS
5. Geofence breach detection occurs within one GPS update cycle of the breach
6. Weather-triggered evacuation alerts reach all affected guests within 120 seconds
7. Operations dashboard (web-ops-dashboard) displays live guest positions on a map with refresh rate matching tracking frequency
8. All tracking data is retained for 90 days for insurance compliance, then archived or purged
9. Unknown adventure categories default to Pattern 3 tracking frequency (10s) per ADR-005 safety-first principle
10. Tracking sessions terminate automatically when the adventure completes
