## Summary

Implement adventure category classification for check-in UI pattern determination.

## Description

### Problem

NovaTrek currently uses the `activity_type` field on trip reservations to determine which check-in experience a guest receives at arrival. There are 10 activity types (HIKING, CLIMBING, KAYAKING, RAFTING, CAMPING, MOUNTAIN_BIKING, CANYONEERING, SNOWSHOEING, SKIING, SAFARI), but these are far too coarse to properly classify the check-in requirements.

For example, a guest booked for backcountry camping currently receives the same basic check-in flow as a guest booked for a simple day hike. The backcountry camper needs gear verification, a safety briefing, guide meetup coordination, transport logistics, and a medical clearance check. The day hiker just needs a confirmation screen.

Similarly, a flatwater kayak outing on a calm lake is treated the same as a whitewater kayak expedition, even though the latter requires specialized gear pickup, safety equipment verification, and a mandatory safety briefing.

### Current Behavior

The `svc-check-in` service reads `activity_type` from the reservation and applies a simple two-tier check-in flow:
- "Simple" activities (HIKING, SAFARI) get a basic confirmation screen
- "Complex" activities (everything else) get full gear pickup + guide meetup

This binary classification is inadequate. Guest feedback consistently reports that day hikers are confused by unnecessary gear steps, while multi-day expedition guests feel the check-in is insufficient for the complexity of their trip.

### Desired Outcome

Introduce an **adventure category classification** system that maps 25 granular adventure categories to 3 distinct check-in UI patterns. The system should also support **booking source overrides** where certain booking channels always receive a specific pattern regardless of adventure category.

### Check-in UI Patterns

| Pattern | Name         | Check-in Steps                                                                |
|---------|--------------|-------------------------------------------------------------------------------|
| 1       | Basic        | Confirmation only. No gear pickup, no guide meetup.                           |
| 2       | Gear Required| Confirmation + gear pickup verification + safety briefing.                    |
| 3       | Full Service | Confirmation + gear pickup + guide meetup + transport coordination + medical. |

## Acceptance Criteria

1. The system SHALL classify each reservation into one of 3 check-in UI patterns based on the `adventure_category` field.
2. All 25 defined adventure categories SHALL have an explicit mapping to a check-in pattern in the classification configuration.
3. If the `adventure_category` is null, empty, or not found in the classification table, the system SHALL default to Pattern 3 (Full Service).
4. Booking source override rules SHALL take precedence over category classification when applicable.
5. The classification mapping SHALL be configuration-driven (not hardcoded), loadable from Spring Cloud Config.
6. The `POST /check-ins` response SHALL include a `determined_pattern` field indicating which pattern was applied.
7. Classification lookups SHALL be cached with a configurable TTL to avoid repeated config reads.
8. The `svc-trip-catalog` Trip schema SHALL be extended with the `adventure_category` field.

Good questions. Based on UX research and operational feedback, we think 3 patterns are the right balance:

- **Pattern 2 (Gear Required)**: Confirmation plus a gear pickup verification step and safety briefing. For mid-complexity activities where specialized equipment is involved.
- **Pattern 3 (Full Service)**: The full works - confirmation, gear pickup, guide meetup, transport coordination, and medical clearance. For high-complexity, multi-day, or high-risk activities.

### Comment 4 - Alex Rivera (Solution Architect) - 2024-08-13 09:45

### Comment 5 - Morgan Chen (Product Owner) - 2024-08-13 10:12

Yes, two important overrides:
- **PARTNER_API** bookings should always get Pattern 1, because our partners handle gear distribution and guide coordination on their end. We just need to confirm check-in.
- **WALK_IN** bookings should always get Pattern 3, because walk-in guests have not gone through any pre-trip preparation and need the full onboarding experience regardless of what activity they sign up for.

All other booking sources (WEBSITE, MOBILE_APP, CALL_CENTER) should use the standard category classification.

### Comment 6 - Jamie Park (Backend Developer) - 2024-08-14 08:30

Architecture question: where will the category-to-pattern mapping data live? Should we create a database table for this so operations can edit it through an admin UI, or is this better as a configuration file?

### Comment 7 - Alex Rivera (Solution Architect) - 2024-08-14 10:15

I recommend a **configuration-driven approach** rather than database-driven. Here is my reasoning:

1. The mapping is relatively stable - adventure categories do not change frequently.
2. A config file (YAML in Spring Cloud Config) can be version-controlled and reviewed through normal PR processes.
3. Config-driven avoids introducing a new database table, admin UI, and the associated maintenance.
4. We can use Spring Cache to load the config once and refresh on a TTL, giving us near-zero performance cost.

If the mapping starts changing frequently (more than once per sprint), we can revisit and move to a database-backed approach.
