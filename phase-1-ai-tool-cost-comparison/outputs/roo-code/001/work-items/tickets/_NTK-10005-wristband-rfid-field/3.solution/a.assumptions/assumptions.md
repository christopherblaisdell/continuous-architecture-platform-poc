# NTK-10005 - Assumptions

## Assumptions

| ID | Assumption | Risk if Wrong | Status |
|----|------------|---------------|--------|
| A1 | The RFID tag format is a hexadecimal string between 8 and 16 characters matching the regex `^[A-F0-9]{8,16}$`. This is derived from the ticket acceptance criteria and the JIRA ticket description. | Validation logic would need to be updated. The kiosk firmware RFID output format may differ from the assumed pattern. Coordinate with the hardware team before finalizing. | Open |
| A2 | RFID uniqueness is scoped to active check-ins only (check-ins with status not in COMPLETED or FAILED). Once a check-in is completed or cancelled, the RFID tag value is released and can be reassigned to a new guest or check-in. | If global uniqueness is required across all historical records, the database constraint and lookup logic become more complex and wristband reuse across days becomes impossible. | Open |
| A3 | The `rfid_tag` field is optional (nullable). Adventures that do not use RFID wristband tracking will continue to function with `rfid_tag` set to null. | If RFID becomes mandatory in a future phase, a data migration for existing check-in records would be needed. The ticket explicitly states the field is nullable, so this assumption is well-grounded. | Confirmed |
| A4 | Downstream services consuming check-in domain events (such as svc-trail-management and svc-guest-experience) use a tolerant reader pattern and will safely ignore the new `rfid_tag` field until they are ready to consume it. No immediate changes are required in downstream services. | If any downstream consumer uses strict schema validation, the new field could cause deserialization failures. This risk is mitigated by the tolerant reader convention documented in the NovaTrek architecture standards. | Open |
| A5 | Kiosk firmware supporting RFID scanning will ship in May 2026, per the ticket reporter (Jordan Lee). The API schema change must be deployed before the firmware release so kiosks have an API field to send the scanned RFID data to. | If the API deployment is delayed past the firmware release date, kiosks with new firmware will have no backend support for RFID data, and scanned tags will be lost. | Confirmed |
| A6 | The `rfid_tag` field should NOT be added to the `CheckInCreate` schema. The RFID tag is captured during the wristband assignment step of the check-in workflow, not at initial check-in creation. This aligns with the current workflow where wristband assignment is a separate step after check-in initiation. | If the workflow changes to capture RFID at initial check-in creation (e.g., scan-to-start), the `CheckInCreate` schema would also need updating. | Open |
