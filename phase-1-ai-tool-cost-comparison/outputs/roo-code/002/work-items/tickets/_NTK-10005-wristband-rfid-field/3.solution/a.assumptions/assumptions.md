# NTK-10005 - Assumptions

## Assumptions

| ID | Assumption | Risk if Wrong | Status |
|----|------------|---------------|--------|
| A1 | The RFID tag format is a hexadecimal string between 8 and 16 characters, matching regex `^[A-F0-9]{8,16}$`. This is derived from the ticket acceptance criteria and the existing `WristbandAssignment` example value `RFID-A3F7B201`. | Validation logic would need updating. The kiosk firmware may produce a different format (e.g., with hyphens, lowercase, or different length). Must be confirmed with the hardware team before implementation. | Open |
| A2 | RFID tag uniqueness is scoped to active check-ins only (status not in COMPLETED, FAILED, CANCELLED). Once a check-in is completed or cancelled, the RFID tag can be reassigned to a new guest on a different adventure. | If global uniqueness is required, the database constraint and lookup logic become more complex. Wristband reuse across adventures within the same day would be blocked. | Open |
| A3 | The `rfid_tag` field is optional (nullable). Adventures that do not use wristband tracking will continue to work without providing an RFID tag. This is consistent with the ticket description stating the field is "optional" and "nullable for adventures that don't use wristband tracking." | If RFID becomes mandatory for all adventures, a data migration would be needed for existing check-in records, and the check-in flow would need to enforce RFID capture before proceeding. | Confirmed |
| A4 | Downstream services consuming check-in events (svc-guest-experience, svc-trail-management) use tolerant reader patterns and will safely ignore the new `rfid_tag` field until they implement RFID-based features. No changes are required from downstream teams for this release. | If a downstream consumer has strict schema validation that rejects unknown fields, the new event field could cause processing failures. This should be verified. | Open |
| A5 | The kiosk firmware supporting RFID scanning ships in May 2026 (per Comment 1 in the ticket report). The API schema change must be deployed before the firmware release so kiosks have a backend field to send scanned tag data to. | If the API deployment is delayed past the firmware release date, kiosks with RFID scanning capability will have no backend support, and the scanned data would be lost. | Confirmed |
| A6 | The `rfid_tag` field on the top-level `CheckIn` schema is populated from the `WristbandAssignment` sub-record when a wristband is assigned. It is NOT provided during the initial `POST /check-ins` call. The `CheckInCreate` schema does not include `rfid_tag`. | If the RFID tag needs to be captured at initial check-in creation (before wristband assignment), the `CheckInCreate` schema would also need modification. | Open |
