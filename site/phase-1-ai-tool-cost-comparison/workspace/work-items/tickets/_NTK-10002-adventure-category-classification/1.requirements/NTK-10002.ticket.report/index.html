    NTK-10002: Implement Adventure Category Classification for Check-in UI Pattern Determination¶

    Summary¶

    Implement adventure category classification for check-in UI pattern determination.

    Description¶

    Problem¶

    NovaTrek currently uses the activity_type field on trip reservations to determine which check-in experience a guest receives at arrival. There are 10 activity types (HIKING, CLIMBING, KAYAKING, RAFTING, CAMPING, MOUNTAIN_BIKING, CANYONEERING, SNOWSHOEING, SKIING, SAFARI), but these are far too coarse to properly classify the check-in requirements.

    For example, a guest booked for backcountry camping currently receives the same basic check-in flow as a guest booked for a simple day hike. The backcountry camper needs gear verification, a safety briefing, guide meetup coordination, transport logistics, and a medical clearance check. The day hiker just needs a confirmation screen.

    Similarly, a flatwater kayak outing on a calm lake is treated the same as a whitewater kayak expedition, even though the latter requires specialized gear pickup, safety equipment verification, and a mandatory safety briefing.

    Current Behavior¶

    The svc-check-in service reads activity_type from the reservation and applies a simple two-tier check-in flow: - "Simple" activities (HIKING, SAFARI) get a basic confirmation screen - "Complex" activities (everything else) get full gear pickup + guide meetup

    This binary classification is inadequate. Guest feedback consistently reports that day hikers are confused by unnecessary gear steps, while multi-day expedition guests feel the check-in is insufficient for the complexity of their trip.

    Desired Outcome¶

    Introduce an adventure category classification system that maps 25 granular adventure categories to 3 distinct check-in UI patterns. The system should also support booking source overrides where certain booking channels always receive a specific pattern regardless of adventure category.

    Check-in UI Patterns¶

    Pattern Name Check-in Steps
    1 Basic Confirmation only. No gear pickup, no guide meetup.
    2 Gear Required Confirmation + gear pickup verification + safety briefing.
    3 Full Service Confirmation + gear pickup + guide meetup + transport coordination + medical.

    Booking Source Override Rules¶

    Booking Source Override Pattern Reason
    PARTNER_API Pattern 1 Partners handle their own gear/guide coordination
    WALK_IN Pattern 3 Walk-ins need full onboarding and safety review
    WEBSITE (use category) Standard classification applies
    MOBILE_APP (use category) Standard classification applies
    CALL_CENTER (use category) Standard classification applies

    Acceptance Criteria¶

    1. The system SHALL classify each reservation into one of 3 check-in UI patterns based on the adventure_category field.
    2. All 25 defined adventure categories SHALL have an explicit mapping to a check-in pattern in the classification configuration.
    3. If the adventure_category is null, empty, or not found in the classification table, the system SHALL default to Pattern 3 (Full Service).
    4. Booking source override rules SHALL take precedence over category classification when applicable.
    5. The classification mapping SHALL be configuration-driven (not hardcoded), loadable from Spring Cloud Config.
    6. The POST /check-ins response SHALL include a determined_pattern field indicating which pattern was applied.
    7. Classification lookups SHALL be cached with a configurable TTL to avoid repeated config reads.
    8. The svc-trip-catalog Trip schema SHALL be extended with the adventure_category field.

    Comment 1 - Morgan Chen (Product Owner) - 2024-08-12 09:14¶

    We have been receiving consistent guest feedback that the check-in experience does not match the complexity of their booked trip. Day hikers are seeing gear pickup screens they do not need, and multi-day expedition guests are not getting the full onboarding they require. The root cause is that we only use activity_type to drive check-in behavior, and there are only 10 activity types for what is really a much wider spectrum of trip complexity.

    Comment 2 - Alex Rivera (Solution Architect) - 2024-08-12 11:30¶

    Morgan, thanks for raising this. A few clarifying questions:

    1. How many distinct check-in experiences do we want to support? Are we thinking a unique flow per category, or a smaller number of patterns that categories map to?
    2. Do we have UX research on what steps each pattern should include?
    3. Are there any exceptions where the pattern should NOT be based on the adventure category (e.g., special booking channels)?

    Comment 3 - Morgan Chen (Product Owner) - 2024-08-12 14:02¶

    Good questions. Based on UX research and operational feedback, we think 3 patterns are the right balance:

    • Pattern 1 (Basic): Just a confirmation screen. For low-complexity activities like day hikes, bouldering, resort skiing, birding safaris.
    • Pattern 2 (Gear Required): Confirmation plus a gear pickup verification step and safety briefing. For mid-complexity activities where specialized equipment is involved.
    • Pattern 3 (Full Service): The full works - confirmation, gear pickup, guide meetup, transport coordination, and medical clearance. For high-complexity, multi-day, or high-risk activities.

    Comment 4 - Alex Rivera (Solution Architect) - 2024-08-13 09:45¶

    That makes sense. One more question - are there situations where the booking SOURCE should override the category-based classification? For instance, do partner-referred guests go through a different flow?

    Comment 5 - Morgan Chen (Product Owner) - 2024-08-13 10:12¶

    Yes, two important overrides: - PARTNER_API bookings should always get Pattern 1, because our partners handle gear distribution and guide coordination on their end. We just need to confirm check-in. - WALK_IN bookings should always get Pattern 3, because walk-in guests have not gone through any pre-trip preparation and need the full onboarding experience regardless of what activity they sign up for.

    All other booking sources (WEBSITE, MOBILE_APP, CALL_CENTER) should use the standard category classification.

    Comment 6 - Jamie Park (Backend Developer) - 2024-08-14 08:30¶

    Architecture question: where will the category-to-pattern mapping data live? Should we create a database table for this so operations can edit it through an admin UI, or is this better as a configuration file?

    Comment 7 - Alex Rivera (Solution Architect) - 2024-08-14 10:15¶

    I recommend a configuration-driven approach rather than database-driven. Here is my reasoning:

    1. The mapping is relatively stable - adventure categories do not change frequently.
    2. A config file (YAML in Spring Cloud Config) can be version-controlled and reviewed through normal PR processes.
    3. Config-driven avoids introducing a new database table, admin UI, and the associated maintenance.
    4. We can use Spring Cache to load the config once and refresh on a TTL, giving us near-zero performance cost.

    If the mapping starts changing frequently (more than once per sprint), we can revisit and move to a database-backed approach.

    Comment 8 - Priya Sharma (UX Designer) - 2024-08-15 11:00¶

    I have shared wireframes for all 3 patterns in Confluence. Quick summary:

    • Pattern 1: Single screen, confirmation button, estimated check-in time: 30 seconds.
    • Pattern 2: 3-screen flow (confirmation, gear checklist with barcode scan, safety briefing acknowledgment). Estimated time: 3-5 minutes.
    • Pattern 3: 6-screen flow (confirmation, gear checklist, guide introduction with photo, transport details with map, medical clearance form, final summary). Estimated time: 8-12 minutes.

    Comment 9 - Alex Rivera (Solution Architect) - 2024-08-16 09:20¶

    Looking at the category list, I want to flag a question. Should some of these be split further? For example, OVERNIGHT_HIKE could range from a well-maintained trail with shelters to a remote wilderness route. Should we have subcategories?

    Comment 10 - Morgan Chen (Product Owner) - 2024-08-16 10:45¶

    Good question, but I think 25 categories is sufficient for this iteration. The 3 check-in patterns are broad enough that minor variations within a category do not change the check-in requirements. If we validate this approach and find edge cases, we can add more categories later. Let us not over-engineer this.

    Comment 11 - Taylor Kim (QA Engineer) - 2024-08-19 08:50¶

    What happens when the adventure_category field is null or contains an unrecognized value? This could happen with legacy bookings that predate this feature, or if a new category is added to the catalog but not yet added to the classification config.

    Comment 12 - Alex Rivera (Solution Architect) - 2024-08-19 10:30¶

    Great edge case catch. I propose we default to Pattern 3 (Full Service) for any null, empty, or unrecognized category. The reasoning is safety-first: it is better to over-service a guest (give them extra steps they might not need) than to under-service them (skip gear verification for someone who actually needs it). The extra steps add a few minutes at worst, but missing a safety briefing for a high-risk activity could have serious consequences.

    Comment 13 - Morgan Chen (Product Owner) - 2024-08-19 11:15¶

    Agreed. Default to Pattern 3 for unknowns. Safety over convenience. Please document this as a formal decision in the solution artifacts.

    Comment 14 - Alex Rivera (Solution Architect) - 2024-08-22 14:00¶

    I have published the initial solution design (v1.0) to Confluence: [NTK-10002 Solution Design v1.0]. It covers the classification table, pattern definitions, and component impacts. Please review and provide feedback.

    Comment 15 - Sam Torres (Dev Lead) - 2024-08-23 09:30¶

    Reviewed the solution design. A few questions:

    1. The classification lookup happens on every check-in request. Have we considered the performance impact? Even with caching, we are adding a lookup step to a latency-sensitive flow.
    2. What is the cache TTL? If we change the config, how long until check-in reflects the update?
    3. Should the classification result be stored on the reservation so subsequent lookups do not need to re-classify?

    Comment 16 - Alex Rivera (Solution Architect) - 2024-08-26 11:00¶

    Good feedback, Sam. I have updated to v1.2 addressing performance:

    1. The classification config is loaded into an in-memory map at service startup and refreshed via Spring Cloud Config refresh events. No per-request I/O.
    2. Cache TTL is configurable, defaulting to 5 minutes. A /actuator/refresh endpoint can force immediate reload.
    3. Storing the classification on the reservation is a good optimization but adds write-path complexity. I recommend we implement without it first and add caching at the reservation level only if we observe performance issues.

    Comment 17 - Alex Rivera (Solution Architect) - 2024-08-29 15:30¶

    Updated to v1.4 with the booking source override table and decision flow diagram. The override check happens BEFORE the category classification lookup, so overridden bookings skip the classification entirely. This simplifies the logic and avoids unnecessary lookups for PARTNER_API and WALK_IN bookings.

    Also updated to v1.6 with minor editorial corrections and final review comments incorporated.
