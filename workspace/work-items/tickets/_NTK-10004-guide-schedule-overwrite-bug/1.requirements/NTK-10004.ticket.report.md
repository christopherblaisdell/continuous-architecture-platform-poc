## Description

Guides have reported that their manually entered availability exceptions, training day blocks, and special notes are disappearing from the system. These manual adjustments are entered through the Guide Portal (backed by svc-guide-management) and include:

- Vacation and PTO blocks
- Training and certification days
- Max-group-size overrides for specific dates
- Special notes (e.g., "No high-altitude trails until medical clearance")

After svc-scheduling-orchestrator runs its nightly optimization cycle, or when an on-demand optimization is triggered by a region manager, these manual entries are gone. Guides must re-enter them, and some are not aware they were lost until they receive an assignment that conflicts with their availability.

## Acceptance Criteria

1. Manual guide adjustments MUST survive schedule optimization cycles
2. No data entered via svc-guide-management should be silently discarded by any other service
3. Concurrent optimization runs across regions must not corrupt guide schedules
4. An audit trail must exist showing what changed a guide's schedule and when

## Steps to Reproduce

1. As a guide, log into Guide Portal and add a vacation block for March 10-14
2. Add a note: "Completing wilderness first aid recertification March 12"
3. As a region manager, trigger schedule optimization for the guide's region
4. As the guide, check the Guide Portal - vacation block and note are gone
5. Guide now appears as available and may receive trail assignments for March 10-14

## Environment

- Production (all regions affected)
- First reported: Cascadia region, Feb 14
- Subsequently confirmed: Sierra and Appalachian regions

### Comment 4 - Morgan Rivera (Solution Architect) - 2026-02-18 11:30

Initial findings from source code review of svc-scheduling-orchestrator. The `GuideScheduleTransformer` class in the optimization pipeline reads the current schedule, runs it through an optimization algorithm, and then writes the result back using a full PUT to `svc-guide-management /api/v1/guides/{guideId}/schedule`. The PUT replaces the entire schedule document, including fields the orchestrator doesn't know about (manual notes, availability exceptions, etc.).

The orchestrator only models these fields in its internal `OptimizedGuideSchedule` DTO:
- `assignedTrails[]`
- `regionId`
- `shiftPattern`
- `effectiveDate`

Fields like `availabilityExceptions`, `manualNotes`, and `maxGroupSizeOverrides` are not in the DTO and are therefore dropped during transformation.

### Comment 5 - Morgan Rivera (Solution Architect) - 2026-02-19 16:05

Confirmed via Elastic logs that svc-guide-management exposes both PUT and PATCH endpoints for guide schedules:

- `PUT /api/v1/guides/{guideId}/schedule` - Full replacement (what orchestrator uses)
- `PATCH /api/v1/guides/{guideId}/schedule` - Partial update (available but unused by orchestrator)

The PATCH endpoint would allow the orchestrator to update only the fields it owns without touching manually-managed fields.

### Comment 6 - Morgan Rivera (Solution Architect) - 2026-02-21 09:40

Found a secondary issue: when multiple regions optimize simultaneously (e.g., nightly batch), guides assigned to cross-region trails can have their schedules written by two optimization processes concurrently. There is no revision control or optimistic locking on the schedule document. Last write wins.

Elastic query for concurrent writes returned 14 instances in the last 30 days where two PUT requests for the same guide occurred within a 500ms window.

### Comment 7 - Sam Patel (Operations Manager) - 2026-02-24 08:55

Third incident confirmed. Appalachian region guide lost certification notes after weekend optimization. This is now affecting guide retention - two guides have complained to HR about having to repeatedly re-enter their information. Can we get an ETA on a fix?

### Comment 8 - Morgan Rivera (Solution Architect) - 2026-02-27 14:20

Root cause is confirmed. Two issues identified:

1. **Primary**: svc-scheduling-orchestrator uses PUT semantics that replace the full schedule document, discarding fields it doesn't model in its DTO. This is a data ownership violation - the orchestrator is overwriting data it doesn't own.
2. **Secondary**: No concurrency control on schedule updates, enabling race conditions during simultaneous regional optimizations.

Next steps: Complete load test validation and draft solution design. Solution will likely involve switching orchestrator to PATCH semantics and adding optimistic locking via document revision fields.
