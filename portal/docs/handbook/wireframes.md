# Wireframes

<p class="subtitle">How to create and update UI wireframes using Excalidraw — and how CI publishes them to the portal</p>

</div>

NovaTrek wireframes are stored as Excalidraw JSON files (`.excalidraw`) in `architecture/wireframes/`. CI converts them to SVG previews, interactive HTML viewers, and Markdown portal pages automatically on every push to `main`.

Architects own the source wireframes. Developers and stakeholders view the rendered output in the portal.

---

## Overview

```
architecture/wireframes/{app}/{name}.excalidraw    ← architect edits this
     │
     ▼ (git push to main)
CI runs generate-wireframe-pages.py
  → portal/docs/applications/{app}/wireframes/{name}.svg   (SVG preview)
  → portal/docs/applications/{app}/wireframes/{name}.html  (interactive viewer)
  → portal/docs/applications/{app}/wireframes/{name}.md    (portal page)
```

## Adding a New Wireframe

### Step 1 — Design in Excalidraw

**Option A — VS Code (recommended for local editing)**

1. Install the Excalidraw extension in VS Code:
   - Open the Extensions panel and search for **Excalidraw** (publisher: `pomdtr.excalidraw-editor`)
2. Create a new file in the appropriate wireframe directory:
   ```
   architecture/wireframes/web-guest-portal/adventure-detail.excalidraw
   ```
3. The file opens in an interactive Excalidraw canvas inside VS Code
4. Design the screen — rectangles, text, arrows, and labels
5. Save the file

**Option B — Excalidraw.com**

1. Go to [excalidraw.com](https://excalidraw.com) — no account required
2. Design the screen
3. Export as JSON: Menu → **Export to file** → select **Excalidraw** format
4. Save the downloaded `.excalidraw` file to the appropriate wireframe directory

### Step 2 — Name the File

Use kebab-case, descriptive names. No version numbers — git history provides versioning.

| Good | Avoid |
|---|---|
| `adventure-detail.excalidraw` | `adventure_detail_v2.excalidraw` |
| `booking-confirmation.excalidraw` | `wireframe-123.excalidraw` |
| `emergency-alert-banner.excalidraw` | `new-screen.excalidraw` |

### Step 3 — Update the Portal Navigation

Open `portal/mkdocs.yml` and add the wireframe page under the appropriate application:

```yaml
nav:
  - Services:
    - Applications:
      - web-guest-portal:
        - applications/web-guest-portal.md
        - Wireframes:
          - Check-in Confirmation: applications/web-guest-portal/wireframes/check-in-confirmation.md
          - Adventure Detail: applications/web-guest-portal/wireframes/adventure-detail.md     # ← new
```

Note: The page file (`portal/docs/applications/web-guest-portal/wireframes/adventure-detail.md`) is generated by CI. You only need to add the nav entry.

### Step 4 — Commit and Push

```bash
git add architecture/wireframes/web-guest-portal/adventure-detail.excalidraw
git add portal/mkdocs.yml
git commit -m "wireframe: add adventure detail screen for web-guest-portal"
git push
```

CI generates the SVG, HTML, and Markdown page automatically.

## Updating an Existing Wireframe

**Option A — Edit the existing file in VS Code**

Open the `.excalidraw` file from `architecture/wireframes/{app}/` directly in VS Code with the Excalidraw extension. Edit and save.

**Option B — Edit online at excalidraw.com**

1. Upload the existing `.excalidraw` file to [excalidraw.com](https://excalidraw.com): Menu → **Open** → select the file
2. Make edits
3. Export as JSON and overwrite the source file in the repository

Then commit and push:

```bash
git add architecture/wireframes/web-guest-portal/check-in-confirmation.excalidraw
git commit -m "wireframe: update check-in confirmation — add wristband assignment step"
git push
```

## Adding Wireframes for a New Application

### Step 1 — Register the application

Add the application to `architecture/metadata/applications.yaml`:

```yaml
web-booking-engine:
  title: Web Booking Engine
  type: web
  description: Public-facing booking engine for guests to search, compare, and reserve adventures
  team: Booking Platform Team
  primary_services:
    - svc-trip-catalog
    - svc-reservations
    - svc-guest-profiles
```

### Step 2 — Create the wireframe directory

```bash
mkdir -p architecture/wireframes/web-booking-engine
```

### Step 3 — Create wireframe files

Add `.excalidraw` files as described in [Adding a New Wireframe](#adding-a-new-wireframe).

### Step 4 — Add to portal navigation

```yaml
nav:
  - Services:
    - Applications:
      - applications/index.md
      # ... existing apps ...
      - web-booking-engine:
        - applications/web-booking-engine.md
        - Wireframes:
          - Adventure Search: applications/web-booking-engine/wireframes/adventure-search.md
```

### Step 5 — Run generators

```bash
bash portal/scripts/generate-all.sh
```

## Wireframe Design Guidelines

These guidelines ensure wireframes are useful for both design collaboration and architecture documentation.

**Use grayscale or low-fidelity styling.** Wireframes communicate layout and flow, not final visual design. Avoid full-color, pixel-perfect mockups — save those for the design system.

**Label every interactive element.** Buttons, links, form fields, and navigation items should have visible labels. A reviewer should understand the user flow without additional explanation.

**Show state transitions.** Where a screen has multiple states (empty state, loading, error, success), use separate wireframe frames or annotation callouts.

**Annotate integration points.** Mark sections that correspond to API calls or real-time data. Example: "Loaded from GET /trips/{id}" near a content panel.

**Link to related architecture.** In the solution design, reference the wireframe file path. In the wireframe page markdown, reference the relevant API endpoints and services.

## When to Create a Wireframe

Wireframes are expected for:

| Scenario | Wireframe Required? |
|---|---|
| New screen or page added to a guest-facing or ops application | Yes |
| Significant layout change to an existing screen | Yes |
| New user flow that spans multiple screens | Yes — one wireframe per screen |
| Backend-only change with no UI impact | No |
| Minor field addition to an existing form | Optional — annotate existing wireframe instead |

For changes that affect guest safety or accessibility (e.g., check-in patterns, emergency alerts), wireframes should be created **before** API contract changes — design the flow first, then define the endpoints.
